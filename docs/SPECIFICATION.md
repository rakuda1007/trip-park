# Trip Park アプリ仕様書

## 1. 概要

### 1.1 アプリの目的

**Trip Park**は、グループで旅行やキャンプを計画・実行するときに、メンバー間で**日程調整**、**情報共有**、**移動計画の共有**、および**立て替え・精算**を一元的に行うためのWebアプリである。家族単位や友人グループなど、複数世帯が参加する想定を前提とする。

### 1.2 解決したい課題

| 課題                                     | 本アプリでの対応                         |
| ---------------------------------------- | ---------------------------------------- |
| 日程のすり合わせがチャットだけだと流れる | 日程調整機能で候補日と回答を集約         |
| 重要なお知らせが埋もれる                 | 掲示板でピン留め・カテゴリ分け           |
| 目的地・経路が口頭や別ツールに分散       | 目的地・経由地・ルートをグループ内で共有 |
| 立て替えの清算が複雑になる               | 支出登録と精算計算（誰が誰にいくら）     |

---

## 2. 想定ユーザー

- **主催者・幹事**: 旅行・キャンプの企画、日程案の提示、掲示板の管理。
- **参加者（メンバー）**: 日程回答、掲示板の閲覧・投稿、移動情報の確認、支出の登録・確認。
- **家族単位**: 精算は「個人」でも「世帯」でも扱えるようにする余地を残す（初期は個人またはグループ内で定義した「支払単位」で設計）。

## 3. 機能一覧

| ID   | 機能名                     | 概要                                               |
| ---- | -------------------------- | -------------------------------------------------- |
| F-01 | ユーザー・アカウント       | ログイン、プロフィール、グループへの参加           |

## 4. 機能詳細

### 4.1 ユーザー・グループ（F-01, F-02）

- **アカウント**: メール／ソーシャルログイン等は実装時に決定。ゲスト参加の要否も任意。
- **グループ**: 「2026年春キャンプ」のように**イベント単位**で作成する想定。メンバーは招待リンクまたは招待コードで参加。
- **権限（例）**:
  - **オーナー**: グループ削除、メンバー除外、設定変更。
  - **管理者**: 掲示板のモデレーション、日程の確定操作。
  - **メンバー**: 各機能の利用（権限により投稿制限あり）。

### 4.2 日程調整（F-03）

- **候補日**: カレンダー上で複数日・複数パターン（「土日」「3連休」など）を登録可能とする。
- **回答**: 各メンバーが「○ / △ / ×」または同等の3段階で回答。未回答のリマインドは将来拡張としてもよい。
- **確定**: 管理者が**確定日**を1つ（または複数行程に分割）選択し、全員に表示。
- **表示**: 一覧表（メンバー × 候補日）で集計が一目で分かる形式を推奨。

### 4.3 掲示板・お知らせ（F-04）

- **投稿**: タイトル、本文、投稿者、投稿日時。
- **カテゴリ例**: 全体連絡 / 持ち物 / 当日の連絡 / その他。
- **重要度**: 通常 / 重要（一覧で強調表示）。
- **ピン留め**: 重要な告知を上部に固定。
- **通知**: 新着投稿のメール・プッシュは任意（MVPではアプリ内表示のみでも可）。

### 4.4 目的地・経由地・移動経路の共有（F-05）

- **目的地**: 名称、住所または地図上の位置、メモ（チェックイン時間、施設URLなど）。
- **経由地**: 車で移動する場合の途中地点（休憩、合流地点など）。順序付きリストで表現。
- **移動経路**:
  - **地図連携**: 外部地図（Google Maps 等）へのリンク、またはアプリ内でルートを表示できると理想的。
  - **共有内容**: 出発地・目的地・経由地をまとめた「旅程ブロック」として保存し、メンバー全員が同じ画面で確認できること。
- **複数ルート**: 行きと帰り、または車両ごとにルートが分かれる場合は、ルートにラベル（例: A車・B車）を付与できるとよい。

### 4.5 支出・精算（F-06）

- **支出登録**: 金額、支払った人（立て替え者）、日付、メモ、カテゴリ（食材、交通、宿泊など）。
- **負担の割り方**:
  - **均等割り**: 参加者全員で均等。
  - **比率指定**: 特定メンバーに重み付け（例: 子どもは0.5）。
  - **除外**: 特定の支出から特定メンバーを除外。
- **精算アルゴリズム**: 各メンバーの「払った額」と「負担すべき額」の差額を求め、**最小回数の送金**で相殺する一般的な精算ロジック（債務グラフの簡約化）を用いる。
- **出力**:
  - 「AさんはBさんに ¥X」「CさんはDさんに ¥Y」のような**送金指示リスト**。
  - 取引履歴のエクスポート（CSV）はあると便利（任意）。

## 5. 画面・ナビゲーション（案）

グループ詳細をホームとし、タブまたはサイドメニューで以下に遷移:

1. ダッシュボード（概要・近日の確定日・新着お知らせ）
2. 日程調整
3. 掲示板
4. 地図・旅程（目的地・経由地・ルート）
5. 精算

---

## 6. 非機能要件（目安）

| 項目         | 内容                                                                                                       |
| ------------ | ---------------------------------------------------------------------------------------------------------- |
| 対応デバイス | スマートフォン優先のレスポンシブWeb（必要に応じてPWA）                                                     |
| 同時利用     | 同一グループ内で複数メンバーが同時編集しても矛盾しにくいよう、日程・支出は楽観的ロックまたは更新日時の表示 |
| プライバシー | グループは非公開。招待されたユーザーのみ閲覧可能                                                           |
| 地図API      | 利用する地図サービスの利用規約・APIキー管理に準拠                                                          |

---

## 7. MVP（最初のリリース）の推奨範囲

優先して実装すると価値が高い範囲の例:

1. グループ作成・招待・メンバー一覧
2. 日程調整（候補・回答・確定）
3. 掲示板（投稿・一覧・ピン）
4. 目的地と経由地のテキスト＋地図リンク（アプリ内ルートは後続でも可）
5. 支出登録と精算結果（均等割りから）
